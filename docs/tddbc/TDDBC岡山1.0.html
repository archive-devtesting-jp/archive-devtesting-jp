TDDBC岡山 1.0

KT(P は無し)

良かったこと

  • ゴールから書くと実装がシンプルになり、書きやすくなる
  • テストの実装方針
  • テストメソッド名を日本語で書く
  • TDD のリズム感が良い
  • ペアプログラミングを体験できたこと
  • TDD を体験できたこと
  • 仕様部分からペアで考えたこと
  • IDEや他の環境の便利さを知れた
  • 同じ仕様のコードをレビューして視点を得たこと
  • ペアプロすることで抜け漏れを気づけた
  • 人によってコードの書き方が全然違うこと
  • private のメソッドに関する見解を聞けたこと

持ち帰りたいこと

  • 正面玄関(public メソッド)からテストする
  • Jenkins を使ってみる
  • TDD を来週から勝手に一人で進めようと思います
  • 仮実装が斬新
  • 皆のその後の収穫が聞きたい
  • テストは全部書かなくても良いこと
  • テストクラスのつくりかた
  • 目標に向かってシンプルに少しずつ作っていく
  • IDEの便利さを伝える(調査重要)
  • 日常業務に取り入れるのは難しくとも、会社として一年に何回か定期的な社内イベントをできるのではないか
  • テストを書くことやリファクタリングの重要さを同僚に伝える
  • さまざまな開発環境を使う
  • TDD をする文化を自社まで持って帰りたい
  • 開発手法を習慣化する
